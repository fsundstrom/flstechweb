Linux Installation

Basic Linux installation is simple with today's versions.  You just boot from the distribution DVD/CD and follow the instructions.  This gives you a running system with the modules you pick.  The trick is setting up the system with what you need and not having too much overhead.

There are 2 basic methodologies to consider when loading a Linux system.  You can just try to install what you need and nothing else.  This is the basis of setting up a production server where you don’t want any overhead or security risk.  For instance, If you are setting up a mail server there is no need to have a web server, GUI interface, Etc.  If you add extra packages you would want to turn them off anyway.  Also with the extra software, you are adding possible security risk that you may need to manage.  If you are running a production server on the internet you will see all kinds of request for added packages like PHP administration services on web servers.  By not loading additional packages you can eliminate many threats.

The other way to go about loading a system is to say you want to load everything.  This is more of a development environment.  Where you don’t want to have to find and load packages that you may need in the future.  In many cases I will start with a system with everything loaded and after the system is working the way I want, I will reload it with just the essentials that I need.

This Web server is a good example of just having what you need.  The whole thing is running on a virtual machine with just a 5Gig hard drive and 1.5 gig of memory.  I do have 2 virtual CPUs assigned as todays operating systems run much better with more than one CPU.  Having more than one CPU helps prevents the system hanging form things like cleanup task or a lot of simtincas hits.

Post install

If you have not created your own install disk or installation server, one of the first things that you will want to do is update all the packages to the latest version.  This is as everyone knows a continuous task.  If you are in a production environment I highly recommend testing all updates first and having a local server or disk to update the production systems from.  The automatic updates are fine but you may have compatibility issues or I have seen updates with added bugs.  So if possible test updates before loading them on important systems!

Getting Updates for Red Hat:

Red Hat uses "yum" to automatically fetch and install packages.  You can also use yum to update all installed packages by running "yum update".  "rpm" is used to look at installed packages or install/remove packages by hand.

Getting Updates for Debian:

Debian uses "apt-get" to automatically fetch and install packages.  To update all installed packages you run "apt-get update" to update the apt db, then run "apt-get upgrade" to upgrade the packages.  To list installed packages or install/remove packages by hand you use "dpkg".

Installing additional software packages

Not all precompiled packages are created equal.  Different Linux distributions use different package managers and they are not compatible with each other.  Red Hat uses rpm (RedHat Package Manager) and Debin uses dpkg (Debian Package).  You also have to be aware of the version of the distribution as not all kernel and library versions are compatible (i.e Red Hat 5 rpm’s and Red Hat 6 rpm’s ) in a few cases this my work but not many.  The last issue here is 64 vs 32 bit systems.  You can no install a 64 bit package on a 32 bit system.  You can install most 32 bit packages on 64 bit systems although this is not generally recommended.

In many cases the package that you are looking for may be on the CD or DVD you installed the system from and the package was just not installed during the installation procedure.  I just installed a Red Hat 6 system and though that everything was installed with the options I picked and nmap was not installed.  So I just installed it from the DVD with “rpm -ivh rpm_file_name.rpm”.  You can also use yum and apt-get to search for and install common packages.

Installing software from source code archives

In many cases a precompiled package may not be available for the Linux version you are running and you will need to compile and install it yourself from the source code.  With todays software this is nominally not that hard.  I am going to go over the most common steps, although I do recommend reading and README or INSTALL files in the archive dir.

Most packages come in the form of a tared zipped archive.  You can just put it in /var/tmp or your home dir.  I recommend a sub dir in your home dir as you may want to go back to it at some time.  Go to the dir that you are going to compile the software in and untar the archive with “tar -zxvf name_of_tar_file.tar.gz”.  cd to the new archive dir and read any install information.  Create the Makefile for your system with “./configure” looking for any errors like missing libraries that you may need to install. Next run make “make” and watch for any errors.  In most cases you install the software with “make install” just like that.  Note that not all packages use the same make commands and sometimes you need to run things like “make all” this is why you need to read the README and INSTALL files.  Good luck ;-)
